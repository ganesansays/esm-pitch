## Password protection

The site has a landing page at `/` that gates access to the deck at `/presentation.html`.

| File | Purpose |
|------|---------|
| `index.html` | Password landing page |
| `presentation.html` | Slide deck (generated by `npm run build`) |
| `js/auth-config.js` | SHA-256 hash of the password |
| `js/auth.js` | Auth helpers (sessionStorage) |

**Change the password:**

```bash
npm run set-password -- "your-new-password"
git add js/auth-config.js && git commit -m "Update site password"
```

### How secure is this?

This is **client-side protection** — good for casual/internal sharing, not for secrets.

- The password hash is visible in `js/auth-config.js` (brute-forceable)
- Determined users can inspect source or bypass with devtools
- Session lasts until the browser tab/session is closed

**For stronger protection, consider:**

| Option | Best for |
|--------|----------|
| **[Cloudflare Access](https://developers.cloudflare.com/cloudflare-one/policies/access/)** (free tier) | Email OTP or SSO in front of GitHub Pages |
| **Netlify Password Protection** | Built-in site-wide password (paid feature) |
| **Private repo + share PDF only** | No public URL at all |
| **Vercel Deployment Protection** | Password or SSO on preview/production |

For executive briefings, **Cloudflare Access** in front of your GitHub Pages domain is the most common upgrade path at zero extra hosting cost.

## Rebuild after adding Stitch exports

1. Add new `code.html` files under `extract-from-stitch/stitch_consulting_presentation_deck/`
2. Update `extract-from-stitch/manifest.json` with the slide order
3. Run `npm run build`

## Host

Upload to Netlify, GitHub Pages, Vercel, or any static host:

```
index.html              ← password landing page
presentation.html       ← slide deck (generated)
js/auth-config.js
js/auth.js
slides/
```

### GitHub Pages

The repo includes a GitHub Actions workflow (`.github/workflows/deploy-pages.yml`) that deploys on push to `main`.

**First-time setup:**

```bash
gh auth login
gh repo create esm-pitch --public --source=. --remote=origin --push
gh api repos/$(gh repo view --json nameWithOwner -q .nameWithOwner)/pages -X POST -f build_type=workflow
```

Your site will be live at `https://<username>.github.io/esm-pitch/` within a minute or two.

**Updates:** push to `main` and the workflow redeploys automatically.

```bash
git push origin main
```
